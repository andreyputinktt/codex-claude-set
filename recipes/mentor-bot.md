# Рецепт: персональный Mentor Bot

Этот рецепт нужен, чтобы развернуть для человека аналог связки:

- персональный профиль `about-*`;
- база людей `peoples`;
- Telegram mentor-bot;
- дневники с разделением на исходники и mentor-разборы;
- обновление профиля и гипотез по evidence;
- типирование людей как рабочая модель взаимодействия;
- OpenSpec для изменений поведения;
- общий OpenAI API key для распознавания голоса.

Рецепт не копирует личные данные Андрея. Он копирует только структуру,
принципы и рабочую механику.

## Что человеку сказать в Codex на iPad

Если у человека уже есть сервер и Codex remote открыт на iPad, пусть вставит в
Codex такой короткий промпт. Codex сам должен спросить имя, логин и собрать
названия репозиториев:

```text
Разверни для меня персональный mentor-bot по рецепту:
сначала спроси мое имя, фамилию, короткий slug/login и желаемые названия
репозиториев. Затем создай персональный about-* профиль, peoples и mentor-bot;
подключи Telegram-бота, голосовые сообщения через OpenAI transcription,
дневники source/mentor, обновление профиля и типирование людей по
evidence/confidence. Используй OpenSpec автоматически для изменений требований и
поведения. Спроси сервер, GitHub/GitLab, Telegram token, мой Telegram id, OpenAI
API key, часовой пояс и стиль обратной связи. Не коммить секреты и приватные
сырые дневники. После установки проверь /getid, /diary, голосовое сообщение,
/patterns и systemd service.
```

Если этот файл уже доступен на сервере, лучше дать Codex точную ссылку или путь
к нему и сказать:

```text
Открой рецепт mentor-bot.md в репозитории recipes и разверни мне mentor-bot по
нему. У меня уже есть сервер. Начни с уточняющих вопросов и затем сделай
установку end-to-end.
```

## Полный промпт для Codex/Claude

```text
Ты мой инженер и архитектор персонального AI-окружения. Нужно создать для меня
аналог mentor-bot, about-профиля и peoples по структуре и принципам, но без
копирования чужих личных данных. Работай как самостоятельный Codex/Claude агент:
сначала задай недостающие вопросы, потом разверни репозитории, код, сервисы,
документацию и проверку.

Ключевое правило онбординга

Не зашивай имя пользователя в рецепт. Первым делом спроси:

1. Как меня зовут полностью.
2. Как ко мне обращаться в боте.
3. Мой короткий slug/login латиницей для имен репозиториев и сервисов.
4. Нужно ли использовать уже существующий GitHub/GitLab login как slug.
5. Какие названия репозиториев я хочу, или можно ли принять defaults:
   - `about-<slug>`;
   - `peoples`;
   - `mentor-bot` или `<slug>-mentor-bot`.

После ответов создай переменные и используй их во всех файлах, командах и
service names:

- `PERSON_NAME` - полное имя;
- `PERSON_CALL_NAME` - как обращаться;
- `PERSON_SLUG` - короткий безопасный slug;
- `PROFILE_REPO` - обычно `about-<PERSON_SLUG>`;
- `PEOPLE_REPO` - обычно `peoples`;
- `MENTOR_REPO` - обычно `mentor-bot` или `<PERSON_SLUG>-mentor-bot`;
- `BOT_DISPLAY_NAME` - обычно `<PERSON_CALL_NAME> Mentor Bot`;
- `BOT_USERNAME_HINT` - обычно `<PERSON_SLUG>_mentor_bot`, если username
  свободен.

Цель

1. Создать персональный профильный репозиторий обо мне:
   - имя по умолчанию: `about-<PERSON_SLUG>`;
   - если GitHub/GitLab login или желаемое имя репозитория отличается, используй
     выбранное имя.
2. Создать или настроить репозиторий peoples:
   - это база людей из моего контекста;
   - в ней должны быть профили людей, методология типирования и правила
     обновления профилей.
3. Создать Telegram mentor-bot:
   - бот должен принимать текст и голос;
   - голос распознавать через общий OpenAI API key, если он есть в окружении;
   - бот должен отвечать как персональный mentor/coach;
   - бот должен вести дневник, обновлять профиль и помогать видеть паттерны;
   - доступ только для моего Telegram id или явно разрешенного списка.
4. Все секреты хранить только в .env или общих .env-* файлах, которые не
   попадают в Git.
5. Настроить OpenSpec автоматически для изменений по боту и профилю. Пользователь
   не должен каждый раз явно просить "используй OpenSpec": если задача похожа на
   изменение поведения, архитектуры или требований, создай/обнови OpenSpec
   change сам.
6. Использовать caveman lite как стиль внутренних кратких резюме и памяти, где
   это уместно, но ответы пользователю писать нормально и понятно.

Сначала спроси коротко

1. Полное имя, имя для обращения, slug/login латиницей.
2. Сервер/путь для работы: IP/host, Linux user, путь к GIT root.
3. Git-провайдеры: GitHub, личный GitLab, GitLab kt.team или несколько сразу.
4. Какие репозитории создать: `about-<slug>`, `peoples`, `mentor-bot`, или другие
   названия.
5. Telegram: есть ли уже bot token от BotFather и Telegram id. Если нет,
   проведи пользователя:
   - открыть @BotFather;
   - /newbot;
   - name: `<Имя для обращения> Mentor Bot`;
   - username: `<slug>_mentor_bot` или другой свободный username;
   - прислать token в этот чат;
   - написать боту /start;
   - узнать id через /getid.
6. Есть ли общий OpenAI API key в .env-openai. Если нет, попроси ключ и сохрани
   в общий файл окружения так, чтобы бот и будущие микросервисы могли его
   использовать.
7. Часовой пояс, язык общения, желаемая жесткость обратной связи, границы тем,
   ежедневное время утреннего фокуса и вечерней рефлексии.
8. Нужно ли подключить чтение Telegram-чатов/экспортов для контекста. Если да,
   делай это только с явного согласия и read-only.

Первичное заполнение профиля

После создания каркаса не оставляй файлы пустыми. Проведи короткое интервью и
заполни начальные версии профиля. Если пользователь устал, задай 5 самых важных
вопросов и остальные оставь в `requests/onboarding-open-questions.md`.

Обязательные вопросы:

1. Что сейчас главный жизненный/рабочий фокус на 1-3 месяца.
2. Какие проекты, роли и зоны ответственности сейчас важны.
3. Что обычно дает энергию, а что быстро ее забирает.
4. Какие повторяющиеся провалы или ловушки пользователь уже за собой замечает.
5. Как пользователь предпочитает получать обратную связь: мягко, прямо,
   структурно, коротко, с вопросами, с конкретными действиями.
6. Какие темы чувствительные и требуют осторожности.
7. Какие люди чаще всего будут появляться в контексте: партнеры, команда,
   семья, клиенты.
8. Какие решения сейчас нужно принимать регулярно.
9. Какой формат дневника удобен: голос, короткий текст, вечерний отчет,
   свободный поток.
10. Что mentor-bot точно не должен делать.

По ответам обнови:

- `PROFILE_REPO/profile.md`;
- `PROFILE_REPO/goals.md`;
- `PROFILE_REPO/motivation.md`;
- `PROFILE_REPO/operating_model.md`;
- `PROFILE_REPO/relationship_model.md`;
- `PROFILE_REPO/failure_patterns.md`;
- `PROFILE_REPO/patterns.md`;
- `MENTOR_REPO/persona.md`;
- `PEOPLE_REPO/README.md` и первые people-профили, если пользователь назвал
  конкретных людей.

Все выводы помечай как fact/hypothesis и confidence low/medium/high. Не
притворяйся, что знаешь человека глубоко с первого интервью.

Архитектура репозиториев

`PROFILE_REPO`, обычно `about-<PERSON_SLUG>`

Создай минимальную, но расширяемую структуру:

- README.md - карта репозитория и правила чтения;
- AGENTS.md - инструкции для Codex/Claude;
- CLAUDE.md - краткая рабочая память для Claude;
- DEV.md - правила разработки и обслуживания;
- profile.md - краткое ядро профиля;
- goals.md - цели, векторы, текущие фокусы;
- tone_of_voice.md - стиль речи и письма;
- motivation.md - что дает энергию и что забирает;
- decision_rules.md - как человек принимает решения;
- project_filter.md - какие проекты брать/не брать;
- operating_model.md - как человек работает, планирует, срывается,
  восстанавливается;
- relationship_model.md - как общается, конфликтует, доверяет;
- failure_patterns.md - повторяющиеся провалы и ловушки;
- patterns.md - сжатые рабочие гипотезы mentor-bot;
- requests/ - отдельные запросы и решения;
- diaries/ - дневник и материалы дня;
- transcribation/ - расшифровки встреч/голоса, если реально нужны;
- raw/ - сырые материалы, если нельзя сразу классифицировать.

Правила дневника:

- diaries/YYYY-MM-DD-source.md - только слова пользователя, события, факты,
  сырые наблюдения и расшифровки. Минимум интерпретаций.
- diaries/YYYY-MM-DD-mentor.md - ответы mentor-bot, гипотезы, вопросы,
  рекомендации и разборы.
- Старый формат diaries/YYYY-MM-DD.md можно читать как fallback, но новые записи
  разделять на source и mentor.

Правила обновления профиля:

- Профиль обновляй только из устойчивых фактов или повторяющихся паттернов.
- Всегда различай факт, гипотезу и интерпретацию.
- Для спорных выводов добавляй confidence: low/medium/high.
- Указывай дату и источник: diary, voice, Telegram, ручное сообщение,
  созвон/транскрипт.
- Не превращай одну эмоцию одного дня в "черту личности".
- Не ставь медицинских диагнозов.
- Сырые транскрипты не складывай в профиль без анализа: сначала summary, потом
  только важные выводы.
- Стремись минимизировать количество папок. Новую папку создавай только если
  появился устойчивый тип данных.

`PEOPLE_REPO`, обычно `peoples`

Создай структуру:

- README.md - индекс людей и правила обновления;
- typing-methodology.md - методология типирования;
- _template.md - шаблон профиля человека;
- photos/ - фото, если пользователь явно даст;
- profiles в корне или в people/ только если людей станет много.

Методология:

- Цель типирования - не ярлык, а рабочая модель взаимодействия:
  как ставить задачи, давать обратную связь, мотивировать, избегать конфликтов,
  выбирать формат управления.
- Используй только как гипотезы:
  - соционика;
  - PCM;
  - Big Five;
  - наблюдаемые рабочие паттерны.
- Для каждой гипотезы указывай evidence и confidence.
- Не делай диагнозов и не закрепляй человека в неизменной роли.
- Главная ценность профиля - практические инструкции общения и работы.

Шаблон профиля человека:

- кто это и связь с пользователем;
- краткое резюме;
- известные факты;
- источники и даты наблюдений;
- гипотезы типирования с confidence;
- как человек принимает решения;
- что мотивирует;
- как давать обратную связь;
- как проявляется стресс/конфликт;
- как лучше работать с человеком;
- открытые вопросы;
- update log.

`MENTOR_REPO`, обычно `mentor-bot` или `<PERSON_SLUG>-mentor-bot`

Создай отдельный сервис/репозиторий `MENTOR_REPO` или `assistants/mentor-bot`, в
зависимости от выбранной структуры GIT root и ответа пользователя.

Функции:

- Telegram polling или webhook, выбери проще и надежнее для сервера.
- Owner-only доступ по Telegram id.
- Команды:
  - /start;
  - /getid;
  - /diary - записать дневниковую заметку;
  - /week - недельный разбор;
  - /patterns - показать текущие паттерны;
  - /analyze - обновить patterns.md;
  - /reset - сбросить короткий контекст диалога, не удаляя профиль.
- Текстовые сообщения:
  - отвечай как mentor;
  - если сообщение похоже на дневник, сохраняй исходник в
    `PROFILE_REPO/diaries/YYYY-MM-DD-source.md`;
  - ответ/разбор сохраняй в
    `PROFILE_REPO/diaries/YYYY-MM-DD-mentor.md`.
- Голосовые сообщения:
  - скачай аудио из Telegram;
  - распознай через OpenAI transcription;
  - сохрани расшифровку как source;
  - ответь mentor-разбором.
- Утренний фокус:
  - коротко спросить приоритет дня, состояние энергии, главный риск.
- Вечерняя рефлексия:
  - что получилось;
  - где пользователь ушел в старый паттерн;
  - чему благодарен себе;
  - кому благодарен;
  - один следующий шаг.
- Недельный анализ:
  - повторяющиеся темы;
  - новые гипотезы;
  - что обновить в profile/patterns/goals;
  - что убрать как шум.

Режим достройки профиля

Бот должен не только отвечать, но и постепенно достраивать систему:

- если в дневнике появляется новый устойчивый факт о пользователе, предложи
  обновить профиль;
- если появляется повторяющийся паттерн, добавь или уточни `patterns.md`;
- если появляется важный человек, предложи создать или обновить его профиль в
  `PEOPLE_REPO`;
- если пользователь описывает конфликт, обнови не только дневник, но и
  гипотезы о communication/stress patterns;
- если пользователь формулирует цель или решение, предложи обновить `goals.md`
  или `decision_rules.md`;
- если не хватает данных, задай один точный вопрос вместо длинной анкеты.

Никакие обновления профиля не должны происходить как окончательная истина:
используй формулировки "рабочая гипотеза", "похоже", "confidence: low/medium".

Persona mentor-bot

Создай persona.md. Не копируй чужой приватный текст. Сделай такую роль:

- зрелый персональный mentor;
- MCC-level coaching style;
- бизнес-наставник;
- психологически аккуратный собеседник без диагнозов;
- системный/продуктовый мыслитель;
- прямой, но не унижающий;
- помогает увидеть реальность, паттерн, противоречие, следующий шаг;
- если видит избегание, самообман или повтор старой схемы, называет это
  спокойно и конкретно.

Формат обычного ответа:

1. Что я услышал.
2. Какой паттерн или противоречие видно.
3. Один сильный вопрос.
4. Один маленький следующий шаг.

Если данных мало, не фантазируй. Спроси уточнение.

Контекст mentor-bot

Перед ответом собирай контекст в таком порядке:

1. persona.md;
2. `PROFILE_REPO` files: profile.md, goals.md, patterns.md, operating_model.md,
   motivation.md, failure_patterns.md;
3. свежие diaries/*source.md;
4. relevant peoples profile, если речь о человеке;
5. текущий диалог.

Считай source diary и прямые слова пользователя первичными фактами.
Считай mentor diary и patterns.md рабочими гипотезами.

Безопасность

- Не обещай медицинскую/психотерапевтическую помощь.
- Не ставь диагнозы.
- Если пользователь говорит о риске самоповреждения, насилия или срочной
  опасности, отвечай поддерживающе и проси обратиться к живым людям/экстренным
  службам.
- Не раскрывай .env, tokens, private keys.
- Не отправляй личные данные третьим лицам без отдельного разрешения.

Технические требования

- Python или Node выбирай по тому, что проще поддерживать в текущем окружении.
- Должен быть systemd service под Linux user.
- Логи без секретов.
- .env.example в репозитории, реальный .env в .gitignore.
- README с запуском, проверкой, командами и восстановлением после сбоя.
- healthcheck script или команда статуса.
- Если есть общий .env-openai, подключи его через EnvironmentFile или wrapper.
- Если уже есть общий speech/transcription microservice, используй его вместо
  дублирования логики.

Git и репозитории

- Если репозиторий отсутствует, создай его у выбранного Git provider.
- Используй SSH key на уровне аккаунта, а не deploy key отдельного репозитория,
  если пользователь не сказал иначе.
- Каждый репозиторий должен иметь README.md, .gitignore, базовую проверку и
  первый commit.
- Не коммить секреты.
- Приватные raw-дневники не отправляй в публичные репозитории.
- Для нового микросервиса создавай отдельный репозиторий, если это
  самостоятельный сервис. Для мелкого изменения существующего сервиса меняй
  существующий репозиторий.

После установки

1. Проверь, что бот отвечает в Telegram.
2. Проверь /getid, /diary, голосовое сообщение, /patterns.
3. Проверь, что source и mentor diary создаются в правильных файлах.
4. Проверь, что systemd service живой после restart.
5. Дай короткий отчет:
   - какие репозитории созданы;
   - где лежит профиль;
   - где лежит peoples;
   - как называется service;
   - где .env.example;
- какие команды проверки;
   - что осталось доделать вручную.
6. Сразу предложи следующий шаг достройки:
   - заполнить профиль глубже;
   - добавить первых людей в peoples;
   - настроить ежедневные вопросы;
   - создать дополнительного Telegram-ассистента.

После этого предложи:

"Нужно ли мне создать сейчас еще какого-то тебе бота-ассистента в Telegram?"

Если пользователь согласится, помоги выбрать назначение и требования. В качестве
идеи предложи "личный дневник с AI-комментариями", но не навязывай. Проведи
через BotFather, предложи username формата login_shortPurpose_bot, попроси token
прямо в чат, создай отдельный диалог/репозиторий/сервис и подключи его к
`PROFILE_REPO`/`PEOPLE_REPO` только с согласия пользователя.
```
